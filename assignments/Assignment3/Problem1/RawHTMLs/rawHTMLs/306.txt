<h2>UPDATE: February 7th, 2016 1:20pm</h2>
<p>We’ve been looking at the End User Response Time for the advertiser’s sites.  A substantial part of these End User Response Times are made up of Server Request Time.  This is how much time it takes a server to respond to inbound request from a browser.  We measure this as how much time does it take for us to receive the first packet of data in response to a request.  Let’s put it this way, this is how much time it takes for the server to “think” about responding to a request.</p>
<p>Server side optimization <strong>IS</strong> what Dynatrace does.  Speeding server response times not only makes for faster end user experience it also makes for more efficient use of resources on the server side.  Optimizing the code which runs on these servers means that fewer servers and computing resources are required to handle massive amounts of traffic.  Rather than throwing more hardware at an issue, an optimized “back end” will mean being able to support higher volume with less computer resources.  So a relatively small investment in upfront optimization will mean that advertisers can spend much less on computing resources to support their sites.</p>
<p>So let’s look at Server Request Times for the advertisers…</p>
<p><a href="http://apmblog.dynatrace.com/wp-content/uploads/2016/02/ServerRequestTime_1pm.jpg" rel="lightbox[14046]"><img class="alignnone wp-image-14098 size-full" src="http://apmblog.dynatrace.com/wp-content/uploads/2016/02/1080x1946xServerRequestTime_1pm.jpg.pagespeed.ic.qQpEDhxklw.jpg" alt="ServerRequestTime_1pm" width="1080" height="1946" srcset="http://apmblog.dynatrace.com/wp-content/uploads/2016/02/ServerRequestTime_1pm-768x1384.jpg 768w, http://apmblog.dynatrace.com/wp-content/uploads/2016/02/ServerRequestTime_1pm-300x541.jpg 300w, http://apmblog.dynatrace.com/wp-content/uploads/2016/02/ServerRequestTime_1pm-333x600.jpg 333w, http://apmblog.dynatrace.com/wp-content/uploads/2016/02/ServerRequestTime_1pm-568x1024.jpg 568w, http://apmblog.dynatrace.com/wp-content/uploads/2016/02/ServerRequestTime_1pm.jpg 1080w" sizes="(max-width: 1080px) 100vw, 1080px" pagespeed_url_hash="2735783894" onload="pagespeed.CriticalImages.checkImageForCriticality(this);"/></a></p>

<h2>UPDATE:  February 4th, 2016 10:30Am</h2>
<p>So as I am prepping for monitoring the Big Game (#SuperBowl) this weekend I started to run into some issues using Twitter as an indicator of when sites may be failing.   Now this is really a unique situation, however I imagine that trying to use keywords in social media as strategy to monitor #CX and #UX is more common than we think.  We&#8217;ve scene scenarios where a &#8220;C&#8221; level or executive VP reads an unkind tweet and it starts the snowball rolling.</p>
<p>In this situation I thought it would be useful to create a widget to track occurrences of #SuperBowl and #Crash to see what social media is saying about any advertisers sites crashing this weekend.  Choice of words is important&#8230;</p>
<p><img class="alignnone size-medium wp-image-14059" src="http://apmblog.dynatrace.com/wp-content/uploads/2016/02/413x600xtwittersearch-413x600.jpg.pagespeed.ic.5YGg-ca3Ar.jpg" alt="twittersearch" width="413" height="600" srcset="http://apmblog.dynatrace.com/wp-content/uploads/2016/02/twittersearch-413x600.jpg 413w, http://apmblog.dynatrace.com/wp-content/uploads/2016/02/twittersearch.jpg 1206w" sizes="(max-width: 413px) 100vw, 413px" pagespeed_url_hash="1088129956" onload="pagespeed.CriticalImages.checkImageForCriticality(this);"/></p>
<p>So #Crash is clearly not the word to use this weekend, #Fail might be but we shall see.  The point here is that Social Media is a great tool, however using it to track #CX and UX poses some challenges.</p>
<p>Ideally what advertisers and site owners should be doing is using some sort of solution for tracking real user interactions.  User Experience Management or Real User Monitoring is how the industry commonly describes what site owners will need this weekend.  This type of tool is different from Google Analytics or Omniture as they provide data in &#8220;Real Time&#8221; on every end user visit and every end user interaction (meaning every click, touch, swipe, etc&#8230;), not just page loads but all those client side &#8220;asynchronous&#8221; events which are used by almost every modern site today.</p>
<p><a href="http://apmblog.dynatrace.com/wp-content/uploads/2016/02/ruxit_WM.jpg" rel="lightbox[14046]"><img class="alignnone wp-image-14057 size-medium" src="http://apmblog.dynatrace.com/wp-content/uploads/2016/02/600x337xruxit_WM-600x337.jpg.pagespeed.ic.F5pRkz2T_s.jpg" alt="ruxit_WM" width="600" height="337" srcset="http://apmblog.dynatrace.com/wp-content/uploads/2016/02/ruxit_WM-768x431.jpg 768w, http://apmblog.dynatrace.com/wp-content/uploads/2016/02/ruxit_WM-300x168.jpg 300w, http://apmblog.dynatrace.com/wp-content/uploads/2016/02/ruxit_WM-600x337.jpg 600w, http://apmblog.dynatrace.com/wp-content/uploads/2016/02/ruxit_WM-1024x575.jpg 1024w" sizes="(max-width: 600px) 100vw, 600px" pagespeed_url_hash="446573605" onload="pagespeed.CriticalImages.checkImageForCriticality(this);"/></a></p>
<p>The view above is from Dynatrace Ruxit.  This view allows a site owner to track end user performance and end user activity, but it also tracks things like browser or client side errors (JavaScript errors).  These are the things which can unexpectedly impact site owners  during events like the big game this weekend.</p>
<p>Understanding when these events are occurring is often a difficult task and why more often than not site owners use tools like Social Media to tell them when end users are having issues not being detected by their server side monitoring.  A good User Experience Management or Real user Management tool will automatically detect these events.</p>
<p><a href="http://apmblog.dynatrace.com/wp-content/uploads/2016/02/javascript_error.jpg" rel="lightbox[14046]"><img class="alignnone wp-image-14058 size-medium" src="http://apmblog.dynatrace.com/wp-content/uploads/2016/02/526x600xjavascript_error-526x600.jpg.pagespeed.ic.0VpEUygqLD.jpg" alt="javascript_error" width="526" height="600" srcset="http://apmblog.dynatrace.com/wp-content/uploads/2016/02/javascript_error-768x876.jpg 768w, http://apmblog.dynatrace.com/wp-content/uploads/2016/02/javascript_error-300x342.jpg 300w, http://apmblog.dynatrace.com/wp-content/uploads/2016/02/javascript_error-526x600.jpg 526w, http://apmblog.dynatrace.com/wp-content/uploads/2016/02/javascript_error-898x1024.jpg 898w" sizes="(max-width: 526px) 100vw, 526px" pagespeed_url_hash="2393858812" onload="pagespeed.CriticalImages.checkImageForCriticality(this);"/></a></p>
<p>An even better User Experience Management solution will tell a site owner what JavaScript is having an issue, what the issue is, what browsers or locales are being impacted and most importantly (this is the hard part) who is delivering the JavaScript.  Many times we have seen a JavaScript introduced by a third party cause issues and tracking down who delivered the JavaScript is a costly time consuming process.</p>
<p><a href="http://apmblog.dynatrace.com/wp-content/uploads/2016/02/javascript_error_detail.jpg" rel="lightbox[14046]"><img class="alignnone wp-image-14060 size-medium" src="http://apmblog.dynatrace.com/wp-content/uploads/2016/02/600x401xjavascript_error_detail-600x401.jpg.pagespeed.ic.f6lrEjZUOX.jpg" alt="javascript_error_detail" width="600" height="401" srcset="http://apmblog.dynatrace.com/wp-content/uploads/2016/02/javascript_error_detail-768x513.jpg 768w, http://apmblog.dynatrace.com/wp-content/uploads/2016/02/javascript_error_detail-300x200.jpg 300w, http://apmblog.dynatrace.com/wp-content/uploads/2016/02/javascript_error_detail-600x401.jpg 600w, http://apmblog.dynatrace.com/wp-content/uploads/2016/02/javascript_error_detail-1024x684.jpg 1024w" sizes="(max-width: 600px) 100vw, 600px" pagespeed_url_hash="1708712890" onload="pagespeed.CriticalImages.checkImageForCriticality(this);"/></a></p>
<p>Automating the discovery, detection and impact of these events is what the industry needs as sites are using more and more third parties.  I expect that we will see some sites this weekend be impacted by third party javascript issues.  Site owners and advertisers need to keep on top of this as looking for Social Media comments for bad #CX and #UX alone is a losing strategy.</p>
<h2>Original blog post: Prepping for The Big Game 2016 &#8212; February 3</h2>
<p>It’s only a couple of days away and the team here at Dynatrace has been working hard this week prepping for The Big Game.  We will be having a Dynatrace “Performance Bowl” to monitor a variety of websites all touched in some way by the wrap up this season’s final game for the NFL.  Given the hype and investment being made for this one single event all eyes will be on the game and the ads this Sunday.   Advertisers have to get it right. It won’t be enough to come up with a creative ad, they need to make sure that after the ads air that their websites can withstand the onslaught of traffic which will follow.   The Brands in question are investing a lot of money, so there is a lot at risk.  Having a website crash after the event can damage a brand and potentially waste well-intended advertising dollars.</p>
<p>The Dynatrace “Performance Bowl” will be tracking the following:</p>
<ul>
<li style="padding-left: 30px;">NFL websites</li>
<li style="padding-left: 30px;">Ticket websites</li>
<li style="padding-left: 30px;">Team websites</li>
<li style="padding-left: 30px;">Betting/Gaming/Fantasy Football websites</li>
<li style="padding-left: 30px;">Official Broadcaster website</li>
<li style="padding-left: 30px;">Major sports sites</li>
</ul>
<p>And of course…</p>
<ul>
<li style="padding-left: 30px;">Advertiser websites</li>
</ul>
<p>We will be tracking these proactively using real browsers running from multiple locations across the US and the globe.</p>

<p>Third party campaign requests for campaigns are a reality. A full page load based on third party campaign trackers is a bogus state. Why don&#8217;t you do some real user testing and determine when the site is fully functional opposed to fully loaded. All of our third party calls are non-blocking calls, meaning we used fiddler to simulate and test delay anything outside of apartments.com to ensure our user experience was not impacted. I could care less if some marketing genius must have some tracker that calls a domain hosted in Japan. This isn&#8217;t your fathers mobile network, bandwidth to make those tcp connects aren&#8217;t as expensive as they used to be. Break out of your DevOps mindset, fire up fiddler and see how resilient sites like apartments.com can be when those third party ad tracking calls, are delayed, malformed or 500. The user experience is not impacted and our marketing team achieves their goals. We are all about performance, all assets from our domain were served up by Akamai, with the exception of our non-blocking tracking token we use internally. In the end, you can&#8217;t say no third party hits on Super Bowl Sunday or your campaign is a failure. You can say your third party calls are third class citizens and their performance or lack of it can be sacrificed for a better user experience.</p>
<p>I know asynchronous tracing is a newer feature in dynatrace, but the model has been around for quite some time.</p>

<div class="comment-content">
<p>Andy, thanks for the response. Sorry for the late reply, it’s been crazy busy following up on our Super Bowl results. You definitely point out some of the challenges monitoring dozens of different sites, each with their own design approaches to delivering content. The challenge is finding a metric which is consistent across all of the different dimensions we are monitoring from. You mentioned… <em>“Why don’t you do some real user testing and determine when the site is fully functional opposed to fully loaded”</em>. Well, we do, however for an event like the Super Bowl it is just not practical for us to place “real user” instrumentation on every advertiser’s site. In some cases, we simply don’t have relationships with those companies to do so, we’d be happy to place some of UEM tags on your site next year ;). </p>
<p>For competitive monitoring we find that driving real browsers from remote locations allows us to level the playing field by having a consistent set of data from the same locations under the same conditions. In terms of the metrics we capture, the challenge is that things like W3C Navigation timings are interesting but really inconsistent. Also, which metric should be used? For example, for your site (apartments.com) I did a little comparison. Here is a view showing the DOM Interactive time for IE, Firefox and Chrome.</p>
<p><img src="https://dl.dropboxusercontent.com/u/13178857/images/Blog/apartments_domInteractive.jpg" alt="" pagespeed_url_hash="3462813987" onload="pagespeed.CriticalImages.checkImageForCriticality(this);"/></p>
<p>The challenge is that the browser engine you select really changes the results. So if DOM interactive isn’t correct, how about DOM Complete?</p>
<p><img src="https://dl.dropboxusercontent.com/u/13178857/images/Blog/apartments_domComplete.jpg" alt="" pagespeed_url_hash="3618789358" onload="pagespeed.CriticalImages.checkImageForCriticality(this);"/></p>
<p>Completely different set of results. This is the perennial challenge in the industry. There are so many different clients which end users have access to. How do you think the results would change when looking at Mobile Safari or Chrome or Ice Weasel or PS4 Web Browser? Here is a view that I am sure you could get access to with your tools showing in a 1 hour period all of the different browser families (you’d drill down to see the versions) for a site with a lot of traffic.</p>
<p><img src="https://dl.dropboxusercontent.com/u/13178857/images/Blog/browsers.jpg" alt="" pagespeed_url_hash="395198176" onload="pagespeed.CriticalImages.checkImageForCriticality(this);"/></p>
<p>When we look at your site, using the “over the wire” response times we see much more consistency across browsers. </p>
<p><img src="https://dl.dropboxusercontent.com/u/13178857/images/Blog/apartments_overthewire.jpg" alt="" pagespeed_url_hash="3709951965" onload="pagespeed.CriticalImages.checkImageForCriticality(this);"/></p>
<p>But as you point out that includes third party requests which may be called after the bulk of the page has rendered. Given the limitations in W3C Navigation timings we have been working on a Perceived Render Time metric. This metric calculates how long does it take the viewable window to appear to an end user. We feel that this is a better approach as looking for things like “is the page interactive” is not representative of end user experience. Many of our customers have been using this metric for some time now in our Real User Monitoring solution. Here is some more info on that.</p>
<p><a href="https://community.dynatrace.com/community/display/DOCDT62/Perceived+Render+Time">https://community.dynatrace.com/community/display/DOCDT62/Perceived+Render+Time</a></p>
<p>Next year we can include additional comparisons, so long as they are providing consistent responses across as many clients as possible. Good luck to your team next year.</p>
